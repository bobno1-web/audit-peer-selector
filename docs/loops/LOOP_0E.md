# Loop 0-E: 비상장 최종 재판정 + 상장 유니버스 정제

> ★ 이 파일이 개발 지시이자 검증 기준이다. (원문 보존)

## 위치
C:\Users\gmg97\Desktop\25-26년\동종업계탐색기_클로드

## 성격
비상장 GO/NO-GO를 마지막으로 한 번 더 판정한다(이번이 끝). 동시에 상장 유니버스 정제.
엔진/채점기 구현은 여전히 금지.

## PART A. 재판정의 정당성
Loop 0-D NO-GO는 두 항목 미달: 종업원수 0.0%(지시서 결함 — ORACLE 어떤 비율에도 안 쓰임),
매출채권 66.7%(진짜 문제). 종업원수를 필수에 넣은 건 지시서 작성자 오류. 이는 "결과 보고 완화"가
아니라 "결과와 무관하게 틀렸던 기준 철회"다(데이터 보기 전에도 참). 재판정은 **1회 한정**.
미달이면 NO-GO 확정, 재시도 없음. DECISIONS D-007에 먼저 기록.

## PART B. 필수 계정 기계적 유도
- B-1. config/default.yaml에 계정-비율 매핑(이미 test_no_leakage 매핑 있으면 재사용).
- B-2. scripts/derive_required_accounts.py: ORACLE ratios → config 매핑 → 필수 계정 집합 유도(사람 타이핑 금지).
  기대: {매출액, 매출총이익, 영업이익, 매출원가, 재고자산, 매출채권}. 총자산·종업원수는 채점 미사용 → 필수 아님.
- B-3. docs/REQUIRED_ACCOUNTS.md 로 출력(스크립트 산출물임을 명시, 손편집 금지).

## PART C. 매출채권 실패 원인 규명 (파서 고치기 전에 먼저)
- C-1. 0-D 매출채권 실패 케이스 전부 특정(재실행 가능, 원자료 보존).
- C-2. 실패 케이스 원문을 실제로 열어 표기 형태 라벨링:
  (a) 합산표기(매출채권및기타채권 등)→파서 (b) 주석에만/본문 없음→파서
  (c) 계정 자체 없음(금융업 등)→데이터 (d) 파일형식(스캔 등)→데이터 (e) 기타.
- C-3. docs/RECEIVABLE_DIAGNOSIS.md: 케이스별 (a)~(e)+근거. 추측 금지. 파서 vs 데이터 비율 명시.
- C-4. (a)/(b)만 파서 개선. (c)/(d)는 실패로 남김. 개선은 계정명 사전 확장으로 하되
  사전은 코드가 아니라 config/account_aliases.yaml 에 둔다.

## PART D. 재측정 — 표본 확대
- D-1. 감사보고서 제출 비상장 200개 무작위(2020~2022).
- D-2. 개선 파서로 필수 6계정 추출률 측정.
- D-3. "성공" = 필수 6계정 전부 확보(관대한 정의 금지). 계정별 + "6개 전부" 둘 다 보고.
- D-4. 결측 편향(n=200).
- D-5. 원자료 보존: runs/2026-XX-XX_spike_unlisted_v2/ 에 summary.json, cases.csv, config.yaml. git 커밋.

## PART E. 최종 GO/NO-GO (기준 결과 전 확정)
GO: 필수 6계정 전부 ≥70% AND 결측 비쏠림. NO-GO: 미달 → PLAN 상장사만 확정 + 한계 명시.
애매하면 NO-GO. 최종. DECISIONS D-008에 "재시도 없음" 명시.

## PART F. 상장 유니버스 정제
- F-1. 35% 결측 정체 규명(ETF/ETN/스팩/리츠, 코넥스, 상장전/상폐, 비12월결산).
- F-2. KRX 상장종목 데이터(증권구분/시장구분/상장일/상폐일) 확보 가능한가 실제로 받아보고 판단.
  가능→정제 사용, 불가→대안+한계 문서화.
- F-3. 생존편향 금지 재확인: 상폐기업은 결측 아님. 상폐일 못 구하면 규칙 못 지킴 → 명시적 보고.
- F-4. 정제 후 상장 확보율 재측정(표본 200).
- 출력: docs/UNIVERSE_DEFINITION.md (정의+구조적 근거, 확보율 실측, 생존편향 대응 가부, 남은 결측).

하드코딩 금지, 키워드 종목분류 금지, "가능해 보인다" 금지, 원인규명 없이 파서개선 금지(C가 D보다 먼저),

# LOOP_PROTOCOL — 루프 실행 프로토콜 (고정)

이 문서는 매 루프를 어떻게 굴리는지 못박는다. Loop 0-A/0-C/0-E에서
**같은 원인의 사고가 3회** 발생했다(지시서 다중화·부재로 인한 오판/미실행).
그 재발을 막기 위한 절차다.

## 실행 순서 (고정)
1. **지시서를 `docs/loops/LOOP_XX.md` 로 먼저 저장**한다. (코드 한 줄 건드리기 전에.)
2. **개발방 실행** — 지시서대로 만든다.
3. **검증방 실행** — 채점 기준은 `LOOP_XX.md`의 완료조건**뿐**이다.
4. **PART Z** — 이번 루프의 모든 발견을 라우팅 표에 따라 반영한다.
5. **`docs/LOOP_LOG.md` 기록** — 비개발자용 한 줄 요약 포함.

## 지시서 단일성 원칙  ★ (사고 3회의 근본 원인)
- 한 루프에 **지시서는 하나뿐**이다.
- 개발방에 넘긴 뒤에는 **그 루프 안에서 지시서를 수정하지 않는다.**
- 바꾸고 싶으면 그 루프를 끝내고 **다음 루프로 넘긴다.**
- 검증 프롬프트의 조건 번호는 `LOOP_XX.md`에서 **복사**한다. 손으로 타이핑하지 않는다.

> 근거: 0-A는 지시서-검증표 불일치로 오판 FAIL, 0-C는 지시서 파일 부재,
> 0-E는 지시서 두 버전 공존으로 일부만 실행됨. 전부 "지시서가 하나가 아니었다"가 원인.

## 발견 → 반영 라우팅 표
| 발견의 성격 | 반영 위치 |
|---|---|
| 지시서·검증 절차 결함 | `docs/VERIFY.md` |
| 설계 판단 | `docs/DECISIONS.md` |
| 프로젝트 규칙 위반 | `CLAUDE.md` + `.claude/rules/` |
| 기계로 막을 수 있는 것 | `.claude/hooks/` + `tests/`  ← **최우선** |
| 반복 작업 절차 | `.claude/skills/` |
| 계획·범위 변경 | `docs/PLAN.md` |
| 데이터 사실 | 해당 조사 문서 |
| 루프 요약 | `docs/LOOP_LOG.md` |

## 철칙
- **모든 발견은 한 곳 이상에 반영**된다. 미반영 발견이 남으면 그 루프는 **미완**이다.
- **기계로 막을 수 있으면 문서에 적지 말고 훅/테스트로 막는다.**
  문서는 안 읽히지만 훅은 안 읽어도 막는다.
- **PART Z는 모든 루프 지시서에 반드시 포함**된다.

## 병렬 실행 프로토콜 (Loop 0-H 3-2)
- **독립 트랙은 병렬화한다** — 서브에이전트 또는 백그라운드 작업으로.
- **가장 오래 걸리는 트랙(대개 데이터 수집)을 먼저 착수한다.** (예: PIT 전량 수집 = 수시간)
- **★ 게이트-종속 트랙은 게이트 PASS 전에 '확정'하지 않는다.** 게이트 PASS 전에는
  수집·적재(되돌릴 수 있는 준비)까지만 하고, 최종 산출물로 승격/확정하는 것은 게이트 뒤로 미룬다.
  게이트 종속 여부는 지시서에서 `[게이트 종속]`으로 표기하고, 검증은 VERIFY 규칙9를 따른다.
- 공유 파일을 동시에 편집하는 트랙은 병렬화하지 않는다(편집 충돌 방지) — 그런 트랙은 순차로.

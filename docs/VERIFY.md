# VERIFY — 검증 규칙

검증방(verifier)이 각 루프의 산출물을 판정할 때 따르는 규칙.

## 규칙
1. **채점 기준을 발명하지 않는다.** 개발 지시서에 명시된 "완료 조건"만 채점한다.
   지시서에 없는 항목으로 FAIL을 내면 그것은 검증방의 오류다.
2. **고치지 않는다. 판정만 한다.** 검증방은 코드를 수정하지 않는다.
3. **지시서와 완료조건이 서로 모순되면**, 코드를 FAIL 내지 말고 "지시서 결함"으로 보고한다.
   (근거: Loop 0-A에서 지시서와 검증표 불일치로 오판 FAIL이 발생했다.)
4. **출력 형식**: 항목별 통과/실패 표 + 마지막 줄에 `PASS` 또는 `FAIL`.
5. **채점 기준은 `docs/loops/LOOP_XX.md` 파일이다.** 그 파일이 없으면 채점하지 않고
   **"지시서 부재"로 반려**한다. 채팅·기억은 근거가 아니다.
   (근거: Loop 0-A는 지시서-검증표 불일치, Loop 0-C는 지시서 파일 부재 —
   같은 원인의 사고가 2회 발생했다.)
6. **검증 요청의 조건 번호가 `LOOP_XX.md`와 다르면**, 채점하지 않고
   **"지시서-검증표 불일치"로 먼저 반려**한다. 검증 프롬프트에 손으로 옮겨적은
   번호를 믿지 말고, 파일의 완료조건과 대조해 불일치를 먼저 잡는다.
   (근거: Loop 0-A / 0-C / 0-E 에서 **동일 원인의 사고가 3회** 발생했다.)
7. **게이트/기준이 대리지표(proxy)면, 직접 증거와 충돌할 때 직접 증거를 우선한다.**
   게이트는 목표(예: 생존편향 없음)를 *직접* 재는 조건과, 그 목표의 *간접* 대리지표를
   섞어 AND 로 걸 수 있다. 직접 조건이 결정적으로 통과했는데 대리지표만 실패하면,
   기준을 결과에 맞춰 바꾸지는 않되(부정행위) **"기준 결함(대리지표 오탐)"으로 보고하고
   사람 판단으로 넘긴다.** 대리지표를 근거로 검증된 방식을 폐기하지 않는다.
   (근거: Loop 0-F PART B — 공시이력 유니버스가 상폐 40/40 제거·확증으로 생존편향 안전임을
   직접 증명했으나, 게이트의 "규모 단조증가 아님"(대리지표)이 실제 시장 성장 때문에 오탐 →
   문자상 FAIL. D-009. **다음 루프 교훈: 게이트는 대리지표가 아니라 직접 메커니즘을 기준으로.**)

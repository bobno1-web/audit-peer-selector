# RECEIVABLE_DIAGNOSIS — 매출채권 추출 실패 원인 규명 (Loop 0-E PART C)

★ 파서를 고치기 전에 원인부터 규명한다. 아래 분류는 **각 실패 케이스의 감사보고서 원문
(document.xml)을 실제로 내려받아 표·텍스트를 검사한 결과**다(추측 아님). 케이스별 기록:
`runs/2026-07-14_spike_unlisted_v2/cases.csv` 의 `recv_class` 열.

## 방법
- 표본 n=200 (감사보고서 제출 비상장). 매출채권 추출을 **기본 별칭(`매출채권`)** 으로 시도.
- 기본 별칭으로 **실패한 54건**을, 원문을 열어 아래로 분류:
  - (a) 다른 표기/합산 — 확장 별칭(수취채권·외상매출금·받을어음 등)으로는 숫자 표에 잡힘 → **파서**
  - (b) 텍스트엔 있으나 숫자 있는 표 행 밖(주석/비표준 표) → **파서(구조)**
  - (c) 계정 자체가 문서에 없음 → **데이터**
  - (d) 파일 형식(스캔 이미지 등) → **데이터**

## 결과 (실측)

| 분류 | 건수 | 비중(실패 54건 중) | 성격 |
|------|:---:|:---:|------|
| (a) 다른 표기/합산 | **6** | 11.1% | 파서 (사전으로 해결됨) |
| (b) 표 밖/주석 | **30** | 55.6% | 파서 (구조 — 사전으로는 미해결) |
| (c) 계정 없음 | **18** | 33.3% | 데이터 |
| (d) 파일 형식 | **0** | 0% | 데이터 |

- **파서 문제 vs 데이터 문제 = 36 : 18 = 66.7% : 33.3%** (기본 별칭 실패 54건 기준).

## 파서 개선(사전 확장)의 실제 효과
- 매출채권 추출률: **기본 73.0% → 확장 별칭 76.0%** (+3.0%p, (a) 6건 회복).
- ★ 한계(정직): C-4가 지시한 "계정명 사전 확장"은 **(a)만** 해결한다.
  (b) 30건은 계정명이 아니라 **값이 파싱 가능한 숫자 표에 없는** 구조 문제라 사전으로는 못 고친다
  (주석 표/병합셀). (c)/(d)는 원천적으로 데이터 문제. → 사전 확장의 상한이 76%인 이유.
- 계정명 사전은 코드가 아니라 `config/account_aliases.yaml` 에 둔다(하드코딩 금지 준수).

## 함의
매출채권은 개선 후에도 76%로, 재무 6계정 중 하위권이다. 실패의 1/3은 데이터에 아예 없는
(부동산·지주·서비스 등) 구조적 결측이라 어떤 파서로도 못 채운다. 이 점이 PART E 최종 판정의
근거 중 하나다(→ `docs/DECISIONS.md` D-008).

# 규칙: 게이트는 구속력이 있다 (gates-are-binding)

## 무엇이 금지인가
게이트가 **FAIL 또는 PENDING**일 때 후속 작업을 시작하는 것.

**게이트를 넘는 사유로 다음을 대지 마라 — 전부 참이어도 사유가 안 된다:**
- "되돌릴 수 있다" (무해)
- "사람이 부재중이다" (무응답)
- "판단상 안전하다" (내가 옳다고 판단함)

**게이트를 넘을 수 있는 유일한 것은 사람의 명시적 승인**이다.
`python scripts/gate.py approve <gate_id> --human-approval <승인>` — 사람만 실행한다.
에이전트는 `create`/`measure`/`fail`/`check` 만 쓴다. **`approve` 는 사람 전용이다.**

## 왜 위험한가
게이트의 존재 이유는 **실패하면 멈추는 것**이다. "판단상 안전"으로 넘을 수 있으면
그건 게이트가 아니라 메모다. 내 판단을 못 믿어서 게이트를 세운 것이다(종업원수·6계정전부·
단조증가 — 매번 "판단상 옳다"고 여겼고 매번 틀렸다).

## 기계적 강제 (문서가 아니라 코드)
- `scripts/gate.py`: PASS 로 가는 유일 경로는 `approve`(사람 승인 인자). create/measure/fail 은
  코드가 PASS 를 거부한다.
- 게이트 파일은 서명(HMAC)된다. **손으로 편집해 status=PASS 로 바꾸면 서명 불일치로 무효** →
  빌드 차단.
- 후속 스크립트는 `gate.require_pass(gate_id)` 로 확인 → 미통과면 `exit 1`.
- 검사: `tests/test_gate_enforcement.py`.

## ★ 측정값을 통제하는 자가 판정을 통제해선 안 된다 (LOOP_0I)
`gate.py judge` 는 **에이전트가 gate 파일에 써넣은 measured 를 믿지 않는다.**
- 각 check 의 `measurement_provenance` 를 따라 **원자료에서 독립적으로 재집계**(`gate_metrics`,
  측정 스크립트와 별개 구현)하고, gate 파일 값과 **대조**한다. 불일치(위조) → 판정 거부.
- 임계 비교는 **재집계값**에 적용한다(에이전트가 쓴 값이 아니라).
- provenance 가 없거나 원자료가 없어 재집계 불가면 → 판정 거부.
- 기계 강제: `tests/test_gate_measurement_integrity.py`(위조·원자료부재 차단 실증).
근거: 0-H에서 "사람 approve 제거 → 코드 판정"이 오히려 **에이전트가 계기판 숫자를 직접 쓰는**
새 구멍을 열었다. 측정 통제와 판정 통제를 분리해 그 구멍을 막는다.

## 위협 모델 (정직히)
에이전트는 FS 접근권이 있어 `.gatekey` 를 읽어 서명을 위조하거나 `gate.py` 자체를 고칠 수도
있다. 그것은 **의도적 위·변조**로, "합리화에 의한 통과"(0-F)와는 다른 종류의 명백한 위반이며
서명·감사·검증방(VERIFY 규칙8)으로 드러난다. 이 게이트가 없애는 것은 **합리화 경로**다:
create/measure/build 어디에도 PASS 가 없으므로, "안전하니 넘자"가 물리적으로 불가능하다.

## 근거 (실제 사고)
- **Loop 0-F**: 생존편향 게이트를 `status=FAIL` 로 기록하고도 PART C~Z 를 진행했다.
  사유 "무응답→최선 판단, 번복가능, 안전 입증". 전부 참이었으나 게이트 위반이었다
  (`DECISIONS` D-011). → 이 규칙과 `gate.py` 로 재발을 코드가 막는다.

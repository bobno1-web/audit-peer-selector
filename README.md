# 동종기업 선정기 (audit-peer-selector)

🇰🇷 한국어 | [🇺🇸 English](README.en.md)

기업 하나를 넣으면 재무적으로 비교가능한 **동종기업(peer)을 순위대로** 뽑아주는 도구. 그리고
**그 도구가 얼마나 잘하는지 자동으로 채점하는 하네스**. 엔진은 갈아치울 수 있고, 채점기는 남는다.

---

## 1. 프로그래밍 소개

초도감사(신규 수임)의 **분석적 절차**에서 감사인은 "이 회사의 재무비율이 동종기업 대비 이상치인가"로
위험을 탐지한다. 그러려면 먼저 **"이 회사를 무엇과 비교할 것인가"**에 답해야 한다. 이 도구는 그 질문에
데이터 기반으로 답한다 — 타겟 하나를 넣으면 비교가능한 peer 목록을 순위로 내고, 각 peer가 왜 뽑혔는지와
얼마나 믿을 만한지를 함께 붙인다.

핵심 차별점:

- **필터가 아니라 다축 유사도 스코어링.** 산업·규모·시가총액·사업내용·성장률을 후보를 탈락시키는
  조건이 아니라 **점수 피처**로 쓴다. 한 단계에서 후보를 버리지 않으므로 경계에 걸친 좋은 peer를 잃지 않는다.
- **산출물에 신뢰등급·근거·확인필요지점 부착.** peer 목록에 신뢰도(HIGH/MEDIUM/LOW), 축별 선정 근거,
  그리고 타겟이 peer 중앙값에서 크게 벗어난 "확인 필요" 지점을 함께 낸다.
- **채점 하네스로 검증된 성능.** "좋은 peer라면 그 peer들로 타겟의 재무비율을 잘 맞힌다"는 원리로,
  정답 라벨 없이 peer 품질을 자동 채점한다. 그래서 개선 여부를 매번 숫자로 확인할 수 있다.

**유니버스:** 국내 상장사 KOSPI + KOSDAQ, 2016~2025년. 비상장·해외는 데이터 근거로 제외했다(사유는
4장·`docs/`). 상장폐지된 기업도 그 시점에 살아있었다면 후보에 포함한다(생존편향 차단).

## 🔍 웹앱으로 바로 써보기 (로컬 웹 UI)

코드를 몰라도 **회사명 하나를 입력창에 넣으면** 그 회사의 동종기업을 순위로 보여주는 로컬 웹앱이 있다.
자매 프로젝트와 같은 스택(로컬 Flask + 정적 HTML)이며, 화면은 확정된 엔진 출력을 **그대로 표시만** 한다 —
순위·유사도 점수·엔진 성능을 바꾸지 않는 **표현 계층**이다. 브랜드 마크는 돋보기(브라우저 탭 favicon·로고 동일 심볼).

**실행법:** `start.bat` 더블클릭 → 필요한 라이브러리를 최초 1회 설치한 뒤 로컬 서버가 뜨고 브라우저가 자동으로
열린다(로컬 전용 `127.0.0.1`, 외부 노출 없음). 입력창에 회사명을 넣고 "시작하기". OpenDART API 키는 **선택**
입력이며, 넣더라도 **이 세션 메모리에만** 두고 파일·로그·디스크에 저장하지 않는다(미리 빌드된 최신 시점
스냅샷에서 순위를 내므로 키 없이도 데모가 동작한다).

![결과 오뚜기](docs/images/web_result_ottogi.png)

- **결과 화면**은 peer 를 두 그룹으로 나눠 보여준다 — **"비교 가능한 동종기업"**(업종·사업내용이 유사)을 위에,
  **"업종은 다르나 규모·시가총액이 유사한 기업"**은 기본 접힌 상태로 아래에 둔다. 유사도 점수는 **절대 기준이 없어**
  (0.46=46% 아님) 화면에 숫자로 표시하지 않고, **위에서부터 비슷한 순서**로만 배열한다.
- 업종이 비슷한 기업을 못 찾으면(예: CJ ENM 같은 복합 대기업) **"찾지 못했습니다"라고 정직하게 안내**하고,
  규모·시총만 유사한 기업만 접이식으로 제공한다 — 억지로 채우지 않는다.

### 이 웹앱이 신경 쓴 것 (정직성)

- **순위를 조작하지 않는다.** 유사도 점수·순위는 엔진 원본 그대로 표시하고, "업종/규모" 그룹은 **시각적 분리**일
  뿐 그룹 안에서 순위를 다시 매기지 않는다.
- **없으면 없다고 말한다.** 업종 유사 peer 가 없으면 채우지 않고 정직하게 안내한다.
- **신뢰등급·근거는 표현 계층이다.** HIGH/MEDIUM/LOW 등급과 축별 근거는 엔진의 예측 성능을 **바꾸지 않는다** —
  결과를 읽기 쉽게 할 뿐이다.
- **최신 데이터에 확정 엔진을 '적용'만 한다.** 웹은 최신 시점(2025-05-15) 스냅샷을 조회하되, dev 에서 확정한
  가중치·임계값을 **재학습 없이 로드해 적용**한다(최신 데이터로 튜닝·재학습하지 않는다). 이 도구의 성능 한계
  (4장의 상한 ≈ median APE 0.50)는 데이터가 최신이어도 동일하다.

## 2. 실행법

### 준비물
- Python 3
- 라이브러리: `numpy`, `pandas`, `pyarrow`(parquet), `PyYAML`, `scikit-learn`(사업내용 텍스트 벡터)
- OpenDART API 키 (재무·공시 취득용)

### OpenDART 키 설정
1. https://opendart.fss.or.kr 에서 API 키를 발급받는다(무료, 40자).
2. `.env.example`를 복사해 `.env`로 만들고 값을 채운다:
   ```
   OPENDART_API_KEY=발급받은_키
   ```
3. 키는 **파일·로그에 저장하지 않는다.** `.env`는 `.gitignore` 대상이고, 코드는 키를 환경변수
   또는 `.env`에서 읽어 API 요청 헤더로만 쓴다(로그 미기록).

### 직접 실행하지 않아도 결과를 볼 수 있다
미리 돌려둔 실제 산출물이 저장소에 들어 있다:
- `runs/2026-07-16_loop8/sample_reports.json` — 완성된 peer 선정 리포트 예시(아래 3장).
- `docs/FINAL_REPORT.md` — out-of-sample 최종 성능·한계·사용 가이드.

### 직접 재현하려면
시점(point-in-time) 데이터층은 용량 때문에 저장소에 포함하지 않는다. 재현은 다음 순서다:

1. **시점 데이터 빌드** (OpenDART 필요, 취득물은 디스크에 캐시되어 재실행 시 재호출 안 함):
   ```
   python scripts/pit_build.py        # 재무·산업·규모 시점 스냅샷
   ```
   사업내용 텍스트·시가총액·성장률·부문 피처는 별도 빌드 스크립트로 채운다(전체 순서는
   `docs/LOOP_LOG.md` 참고).
2. **엔진 실행** → peer 목록 산출(`peers.parquet`):
   ```
   python -m engines.baseline.run      # 산업+규모 baseline
   python -m engines.similarity.run    # 다축 유사도(L6)
   ```
3. **리포트 생성** → 신뢰등급·근거·확인필요지점이 붙은 산출물:
   ```
   python scripts/build_report.py      # → runs/…/sample_reports.json, peer_report.csv, thresholds.json
   ```

> 엔진(`engines/`)과 채점기(`scoring/`)는 서로를 import하지 않는다. 유일한 접점은 `peers.parquet`
> 파일이다("시험지 훔쳐보기" 차단 — 4장).

상위 peer(순위·유사도·축별 근거):

| 순위 | peer | 유사도 | 산업 | 규모 | 시총 | 텍스트 | 성장 |

| 3 | 00445160 | 0.543 | 0.214 | 0.198 | 0.082 | 0.043 | 0.007 |

| 비율 | peer 중앙값 | 타겟 실제 | 이탈 | 확인필요 | 비교가능 |
|---|--:|--:|--:|:--:|:--:|
| 매출총이익률 | 0.208 | 0.112 | 85.2% 하위 | — | ✔ |
| 영업이익률 | 0.057 | 0.043 | 30.8% 하위 | — | ✔ |
| 매출채권회전율 | 4.425 | 3.094 | 43.0% 하위 | — | ✔ |

### 신뢰등급 MEDIUM — 타겟 `00101257` (응집도 0.4902) · "확인 필요" 예시
- 재고자산회전율: peer 중앙값 16.90 vs 타겟 1.96 → **이탈 763%, `확인필요=true`.** 타겟이 peer 대비
  유난히 멀다는 **약한 신호**(사람이 볼 값어치)로 표시된다.

### 비교 부적합 예시 — 타겟 `00100939`
- 매출총이익률: 타겟 실제 0.024로 손익분기 근처 → **`비교가능=false`, note "비교 부적합(손익분기 근처)".**
  분모(실적)가 0 근처라 비율 비교가 구조적으로 부적합함을 알린다(4장 상한 참조).

### 출력 스키마 (요약)
```json
{
  "target": "<기업코드>", "as_of": "YYYY-05-15",
  "peer_confidence": "HIGH | MEDIUM | LOW",   // peer 응집도(top-k 유사도 평균)의 dev 삼분위
  "peers": [ { "rank": 1, "peer_code": "…", "similarity": 0.57,
               "rationale": { "industry":…, "scale":…, "mktcap":…, "text":…, "growth":… } } ],
  "ratios": [ { "ratio": "…", "peer_median": …, "target_actual": …,
                "deviation_pct": …, "check_needed": false, "comparable": true } ]
}
```
범용 CSV(`peer_report.csv`)도 함께 낸다. 전체 스키마는 `docs/OUTPUT_FORMAT.md`.

## 4. 설계 원칙과 넘은 난관 ★ 이 프로젝트의 핵심

여기가 결과보다 중요하다 — **어떻게 사고했는가**. (상세는 `docs/JOURNEY.md`.)

### 제1원칙 — 정답 라벨의 부재를 어떻게 넘었나
"이 기업의 진짜 peer 집합"이라는 정답표는 없다. 시장의 peer 목록(증권사 리포트 등)은 밸류에이션·
공모가 산정 목적으로 오염된 라벨이라 감사용 동종성과 다르다 → **쓰지 않는다.** 대신 **재무비율 예측
오차**로 채점한다: 엔진이 뽑은 peer들의 비율 중앙값을 타겟의 예측치로 삼고, 실제와의 오차(APE)로
peer 품질을 잰다. 이 채점기(오라클)를 먼저 만들어 **동결**하고, 그 뒤에야 엔진을 손댄다 — 점수가
안 나온다고 자를 고치는 자기기만을 원천 차단하기 위해서다.

### 제2원칙 — 필터가 아니라 스코어링
산업·규모로 후보를 걸러내면 한 번 탈락한 후보를 되살릴 수 없다. 경계에 걸친 좋은 peer를 구조적으로
잃는다. 그래서 **전 종목을 점수화·랭킹**한다. 산업·규모·시총·사업내용·성장은 탈락 조건이 아니라 피처다.

### 제3원칙 — 하드코딩·룩어헤드·생존편향 금지 (구조로 강제)
- **하드코딩 금지:** 종목코드·회사명 리터럴, 임의 임계값 상수 금지. 상수(페널티·분리·신뢰등급 경계)는
  전부 dev 데이터 분포에서 유도한다.
- **룩어헤드 금지:** 시점 T의 peer는 T 이전 정보만으로 고른다. 공시는 접수일 기준으로 색인하고,
  T 이후 정정본은 미가용으로 다룬다.
- **생존편향 금지:** T 시점에 상장돼 있던 기업은 지금 상장폐지됐어도 그 시점 후보에 포함한다.

### 넘은 난관 ① — 자기 채점의 함정
엔진이 채점 대상(4비율을 계산할 수 있는 계정)을 볼 수 있으면, "마진 비슷한 5개"를 골라 만점을 받는
커닝이 가능하다. 그래서 엔진 입력을 허용 목록(산업·규모·시총·사업내용·성장)으로 한정하고, 엔진
(`engines/`)과 채점기(`scoring/`)를 **물리적으로 분리**한다(상호 import 금지, 유일 접점은
`peers.parquet` 파일). 나아가 **개발방과 검증방을 분리**해, 검증방은 개발방 코드를 믿지 않고 원자료에서
**직접 재집계**해 대조한다.

### 넘은 난관 ② — 검증 게이트 자체의 설계 오류
데이터 품질 게이트를 여러 번 잘못 설계했다 — 종업원수를 필수 계정에 넣거나, 항진명제(정의상 항상 참)·
순환논증·측정 불가능한 기준을 담는 식으로 **여섯 번**. 그때마다 하네스가 발견했고, 검사 규칙을 문서가
아니라 **코드로** 추가했다("기준이 논리적으로 실패 가능한가"까지 자동 검사). 게이트가 `FAIL`인데도
진행한 위반, 산출물 라벨이 실제와 어긋난 거짓 라벨 사고, 계정 매핑 버그까지 — **숨기지 않고 기록하고
재발을 코드가 막게** 한 과정 자체가 이 프로젝트의 핵심 설계다. (실패는 오라클에 남는 것이 아니라
`docs/DECISIONS.md`에 근본원인과 함께 남는다.)

### 넘은 난관 ③ — 결과를 정직하게 읽기
holdout(2023~2025)을 개발 내내 봉인하고, 모든 설정을 동결(해시 고정)한 뒤 **딱 한 번** 개봉했다.

- **검증된 것:** out-of-sample에서 최종 엔진이 산업+규모 baseline을 **−10% 유의하게** 이겼고
  (firm-clustered 부트스트랩 신뢰구간이 0을 포함하지 않음), dev 순위가 holdout에서 완전히 유지됐다
  (과적합이 아니라 실재하는 개선).
- **미달한 것(숨기지 않는다):** 야심찬 목표선(baseline −20%)은 미달했다. 6개 축·예측방식·신경망
  임베딩을 전부 실측해 도달한 벽이 **median APE ≈ 0.50**이다. 이는 실패가 아니라 **"peer 중앙값으로
  재무비율을 예측하는 방식"의 구조적 상한**의 좌표다 — 남은 오차는 동종그룹 내 사업 이질성이라
  peer 선정을 더 정교화해도 줄지 않는다. **이 한계를 아는 것이 감사 활용의 전제다.**

결론: 이 도구는 "더 나은 peer 선정기 + 이상 탐지기"로는 실용 가능하되(검증됨), 정밀 예측기로는 아니다.
**상한을 실증한 것**이 결과다.
